# ADR 0003 — BFF (bff-web) separado del API Gateway

## Estado
Aceptado

## Contexto
La pantalla de feed de Angular necesita, por publicación, el mensaje (post-service),
el alias del autor (user-service) y el contador de likes (like-service). Sin una
capa de agregación, el frontend tendría que hacer 3 llamadas y ensamblar el
resultado, o los microservicios de dominio tendrían que llamarse entre sí solo por
conveniencia de UI (acoplamiento indebido).

## Decisión
`bff-web` es un microservicio adicional, stateless y sin base de datos propia, cuya
única responsabilidad es orquestar llamadas HTTP en paralelo (WebClient +
Resilience4j) hacia los microservicios de dominio y componer una respuesta ya
optimizada para el frontend. El API Gateway sigue siendo el único punto de entrada
público — el frontend nunca habla directo con `bff-web`; el Gateway enruta
`/bff/**` internamente.

## Consecuencias
- (+) Los microservicios de dominio nunca se llaman entre sí para fines de UI.
- (+) El frontend hace una sola llamada por pantalla compuesta.
- (+) Circuit breaker y fallback viven en un solo lugar (bff-web), no repartidos.
- (-) Un componente más que desplegar, versionar y monitorear.
